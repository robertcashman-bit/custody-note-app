# Custody Note App — Deep Audit Report

**Date:** 17 March 2026  
**Version audited:** 1.4.90  
**Auditor scope:** Full codebase, schema, forms, IPC, views, tests, security

---

## 1. Executive Summary

The app is a well-built, feature-rich Electron custody note tool with encrypted storage, sync, cloud backup, LAA form generation, billing, and auto-update. It handles most custody workflows competently. However, the audit found **4 critical**, **11 high**, **19 medium**, and **12 low** severity issues across data integrity, UX, validation, security, and legal workflow coverage.

The most dangerous findings are:

1. **No `beforeunload` guard** — closing the window mid-edit silently discards unsaved work.
2. **Navigating Home/List/Settings from a dirty form** has no save-or-discard prompt (only "New Attendance" warns).
3. **Soft-deleted records are unrecoverable** — the UI promises "You can restore it later" but provides no restore mechanism.
4. **`offenceSummary` was missing from home/list queries** (already fixed in this session) — similar extraction gaps may exist for other fields used in list cards and widgets.
5. **Billing panel Escape handler leaks** — accumulates on repeated open/close.
6. **Draft deduplication can silently overwrite** an unrelated draft if DSCC or client+date+station collide.

## 4. Confirmed Bugs

### BUG-01: `offenceSummary` missing from home screen queries
- **Severity:** High
- **Confidence:** Confirmed
- **Where:** `main.js` lines 3915–3920 (`attendance-list`), 3929–3951 (`attendance-home-stats`); `app.js` line 3081–3100 (`getHomeStatsDetails`)
- **Scenario:** Home screen Active Matters always shows "Offence not yet entered"
- **Why it matters:** Solicitor can't identify cases at a glance
- **Cause:** Lightweight queries didn't extract `offenceSummary` from JSON data blob; legacy fallback didn't include it
- **Fix:** **Already applied this session** — added `json_extract(data, '$.offenceSummary')` to both queries and added `offenceSummary` to the legacy fallback

### BUG-02: Billing panel Escape handler memory leak
- **Severity:** Medium
- **Confidence:** Confirmed
- **Where:** `renderer/views/billing.js` lines 306–309
- **Scenario:** Open billing panel → close via Close button or backdrop → open again → multiple Escape handlers stacked
- **Why it matters:** Each open adds a new `keydown` listener; only Escape-close removes it; close/backdrop/invoice-create paths don't
- **Cause:** `onKeyDown` is only removed inside its own handler body (`if (e.key === 'Escape')`)
- **Autofix:** Yes — see §10

### BUG-03: Soft-deleted records are unrecoverable
- **Severity:** High
- **Confidence:** Confirmed
- **Where:** `app.js` line 5191 ("You can restore it later if needed"); no restore UI exists
- **Scenario:** User deletes a record, then wants it back
- **Why it matters:** The confirmation message is misleading; data is effectively lost from the UI. Only a DB-level restore or backup restore can retrieve it.
- **Cause:** Soft-delete sets `deleted_at` and `deletion_reason`, but no view or action exposes these records
- **Autofix:** Yes — see §10

### BUG-04: No `beforeunload` guard on window close
- **Severity:** Critical
- **Confidence:** Confirmed
- **Where:** `app.js` — no `beforeunload` listener found anywhere in the codebase
- **Scenario:** User is mid-edit, closes the Electron window → all unsaved changes since last autosave (up to 30s) are silently lost
- **Why it matters:** In urgent custody work, losing even one field of notes is operationally damaging
- **Cause:** Never implemented
- **Autofix:** Yes — see §10

### BUG-05: Navigation away from dirty form has no guard (except "New Attendance")
- **Severity:** Critical
- **Confidence:** Confirmed
- **Where:** `app.js` — `showView('home')`, `showView('list')`, `showView('settings')` called directly from nav without save check
- **Scenario:** User is editing a record → taps Home/Records/Settings in the nav bar → form is abandoned with no warning
- **Why it matters:** Same as BUG-04 — silent data loss
- **Cause:** Only the "New Attendance" nav button checks `currentAttendanceId` and shows a confirm dialog (line 11960)
- **Autofix:** Yes — see §10

### BUG-06: `saveAndExit` deletes record without confirmation if form is empty
- **Severity:** Medium
- **Confidence:** Confirmed
- **Where:** `app.js` lines 8666–8673
- **Scenario:** User opens a draft, clears all fields, clicks Save & Exit → record is silently hard-deleted (no confirmation, no audit)
- **Why it matters:** Accidental data destruction; `hasMeaningfulData` may return false if only non-key fields remain
- **Cause:** Designed for cleanup but uses `attendanceDelete` without user prompt

### BUG-07: Dead code in LAA forms (CRM builders)
- **Severity:** Low
- **Confidence:** Confirmed
- **Where:** `renderer/laa-forms.js` — `buildCRM1`, `buildCRM2`, `buildCRM3`, `buildCRM14Summary` all return a "Strict" variant immediately; remaining code (hundreds of lines) is unreachable
- **Why it matters:** Maintenance confusion; dead code is a liability
- **Cause:** Refactored but old code not removed

## 5. Likely Bugs / Risk Areas

### RISK-01: Draft deduplication overwrites unrelated drafts
- **Severity:** High
- **Confidence:** Likely
- **Where:** `main.js` lines 4139–4151 (`findExistingDraftIdByCaseKey`)
- **Scenario:** Two different clients at the same station on the same date with no DSCC → same case key → second Quick Capture overwrites first
- **Why it matters:** Complete data loss of the first record, with no warning
- **Cause:** Dedup key is `dsccRef || (clientName + date + station)` — too coarse; common for duty solicitors to see multiple clients at one station in one day

### RISK-02: Autosave race with manual save
- **Severity:** Medium
- **Confidence:** Likely
- **Where:** `app.js` — `quietSave` (autosave) and `saveForm` (manual) both call `attendanceSave`; `_draftSaveInFlight` guard exists but `saveForm` doesn't check it
- **Scenario:** Autosave fires while user clicks Finalise → both fire `attendanceSave` concurrently → status inconsistency
- **Cause:** `saveForm` bypasses the `_draftSaveInFlight` guard

### RISK-03: Concurrent save from two devices via sync
- **Severity:** Medium
- **Confidence:** Likely
- **Where:** `main/syncWorker.js` — push/pull logic
- **Scenario:** Two devices edit the same draft and sync → last-write-wins unless conflict detected
- **Cause:** `sync_version` increment is per-device; conflict detection may miss fast successive saves

### RISK-04: `collectCurrentData` reads stale DOM
- **Severity:** Low
- **Confidence:** Possible
- **Where:** `app.js` — `collectCurrentData` reads all input values from the DOM
- **Scenario:** User changes a field but the DOM input hasn't fired its `change` event yet (e.g., date picker with pending commit) → stale value saved
- **Cause:** Reliance on DOM as source of truth rather than reactive state

### RISK-05: Email OIC can open Outlook Web with empty recipient
- **Severity:** Low
- **Confidence:** Likely
- **Where:** `renderer/views/email-modal.js` — OIC modal doesn't validate To field before opening
- **Scenario:** User opens Email OIC with no officer email entered → Outlook Web compose may open with an empty To field
- **Cause:** Quick Email validates but the standard OIC modal does not; compose uses Outlook Web deeplink (not mailto)

### RISK-06: Reports view missing null check on rows
- **Severity:** Low
- **Confidence:** Likely
- **Where:** `renderer/views/reports.js` — uses `rows.forEach` without checking if `rows` is null/undefined
- **Scenario:** `attendanceListFull` fails or returns undefined → crash
- **Cause:** Missing defensive check

### RISK-07: `formData` is a global mutable object
- **Severity:** Medium
- **Confidence:** Confirmed (architecture risk)
- **Where:** `app.js` line 9 — `var formData = {}`
- **Scenario:** Any function anywhere can mutate `formData` at any time; no immutability, no change tracking, no dirty flag
- **Cause:** Architectural choice; mitigated by autosave but creates subtle bugs

## 9. Security / Privacy Risks

### SEC-01: No inactivity auto-lock timeout on the form view
- **Severity:** High
- **Confidence:** Likely
- **Where:** Session lock exists but unclear if it activates when the form has sensitive client data visible
- **Impact:** Solicitor walks away from laptop at police station → sensitive data visible to anyone
- **Fix:** Ensure lock activates within 5 minutes of inactivity when client data is on screen

### SEC-02: `rejectUnauthorized: false` for API calls
- **Severity:** Medium
- **Where:** `main.js` — `_trustedApiAgent` with `rejectUnauthorized: false`
- **Impact:** MITM possible; noted as intentional (AV MITM workaround) but still a risk
- **Mitigation:** Consider certificate pinning for custodynote.com

### SEC-03: Clipboard operations expose sensitive data
- **Severity:** Low
- **Where:** Email modal "Copy Email" puts client details on clipboard; diagnostics export copies to clipboard
- **Impact:** Clipboard contents persist and may be pasted elsewhere
- **Mitigation:** Consider clearing clipboard after a timeout

### SEC-04: PDF/CSV exports saved to Desktop without encryption
- **Severity:** Medium
- **Where:** `main.js` — `save-csv`, `print-to-pdf` save to user-chosen or default Desktop path
- **Impact:** Sensitive legal notes in plaintext on disk, potentially synced to cloud storage
- **Mitigation:** Warn user; consider encrypted export option

### SEC-05: No audit entry when record is re-opened from finalised
- **Severity:** Medium
- **Where:** `app.js` — re-open sets status to draft via `attendanceSave` with `unlock: true`; audit log records "updated" but not the specific "re-opened from finalised" action
- **Impact:** Audit trail doesn't clearly show who un-finalised a record and why
- **Fix:** Add dedicated `reopened` audit action with reason prompt

### SEC-06: Master key fallback uses machine-derived obfuscation
- **Severity:** Medium
- **Where:** `main.js` — `master.fallback` file used when `safeStorage` is unavailable
- **Impact:** If `safeStorage` is unavailable (e.g., headless/CI), the fallback key is weaker
- **Mitigation:** Warn user if running in fallback mode

## 10. Recommended Autofixes

### AUTOFIX-01: Billing panel Escape handler leak

**File:** `renderer/views/billing.js`

```javascript
// BEFORE (line 298–309):
function _bindBillingEvents(recordId, opts) {
  var overlay = document.getElementById('billing-panel-overlay');
  if (!overlay) return;

  overlay.querySelector('.billing-panel-close').addEventListener('click', closeBillingPanel);
  document.getElementById('billing-cancel').addEventListener('click', closeBillingPanel);
  overlay.addEventListener('click', function (e) { if (e.target === overlay) closeBillingPanel(); });

  function onKeyDown(e) {
    if (e.key === 'Escape') { closeBillingPanel(); document.removeEventListener('keydown', onKeyDown); }
  }
  document.addEventListener('keydown', onKeyDown);

// AFTER:
function _bindBillingEvents(recordId, opts) {
  var overlay = document.getElementById('billing-panel-overlay');
  if (!overlay) return;

  function onKeyDown(e) {
    if (e.key === 'Escape') closeBillingPanel();
  }
  document.addEventListener('keydown', onKeyDown);
  // Store handler reference so closeBillingPanel can remove it
  overlay._billingEscHandler = onKeyDown;

  overlay.querySelector('.billing-panel-close').addEventListener('click', closeBillingPanel);
  document.getElementById('billing-cancel').addEventListener('click', closeBillingPanel);
  overlay.addEventListener('click', function (e) { if (e.target === overlay) closeBillingPanel(); });
```

```javascript
// BEFORE (closeBillingPanel, line 580):
function closeBillingPanel() {
  _billingPanelOpen = false;
  var overlay = document.getElementById('billing-panel-overlay');
  if (overlay) overlay.remove();
}

// AFTER:
function closeBillingPanel() {
  _billingPanelOpen = false;
  var overlay = document.getElementById('billing-panel-overlay');
  if (overlay) {
    if (overlay._billingEscHandler) {
      document.removeEventListener('keydown', overlay._billingEscHandler);
    }
    overlay.remove();
  }
}
```

### AUTOFIX-02: Add `beforeunload` guard (app.js)

Add near the end of `app.js`, inside the DOMContentLoaded block:

```javascript
window.addEventListener('beforeunload', function(e) {
  if (!currentAttendanceId) return;
  if (currentRecordStatus === 'finalised') return;
  var data = collectCurrentData();
  if (hasMeaningfulData(data)) {
    e.preventDefault();
    e.returnValue = '';
  }
});
```

### AUTOFIX-03: Navigation guard for dirty form (app.js)

Wrap the navigation handler to check for unsaved work:

```javascript
// In the nav click handler, before showView calls for home/list/settings/firms:
function guardedShowView(name) {
  if (currentAttendanceId && currentRecordStatus !== 'finalised') {
    var data = collectCurrentData();
    if (hasMeaningfulData(data)) {
      showConfirm('You have unsaved changes. Save as draft before leaving?', 'Unsaved Changes', {
        confirmText: 'Save & leave',
        cancelText: 'Discard & leave',
        thirdText: 'Stay'
      }).then(function(result) {
        if (result === 'confirm') {
          window.api.attendanceSave({ id: currentAttendanceId, data: data, status: 'draft' }).then(function() {
            currentAttendanceId = null; stopAutoSave(); showView(name);
          });
        } else if (result === 'cancel') {
          currentAttendanceId = null; stopAutoSave(); showView(name);
        }
        // 'third' or dismiss = stay
      });
      return;
    }
  }
  showView(name);
}
```
